# ALIGNMENT.md — Problem Statement Traceability

**Brief:** *Build a GenAI-enabled solution that enhances stadium operations and the overall
tournament experience for fans, organizers, volunteers, or venue staff. The solution must
leverage Generative AI to improve navigation, crowd management, accessibility, transportation,
sustainability, multilingual assistance, operational intelligence, or real-time decision support
during the FIFA World Cup 2026.*

This file maps every one of the 8 named categories to a specific, runnable feature in
`index.html` / `matchday-ai.js`, the exact GenAI capability it uses, and why a non-AI system
could not do the same job. Every feature listed here makes a **live network call to an
LLM** (`callLLM()` in `matchday-ai.js`), reached through one of two modes configured in
**Settings**: (1) the recommended built-in Express proxy (`server.js`), which keeps
`OPENAI_API_KEY` server-side in `.env` and is never exposed to the browser, or (2) a
bring-your-own-key direct-browser fallback (stored only in `localStorage`) for when the page
is opened via `file://` without a server. Nothing below fabricates an AI response client-side
— if neither mode is configured, the feature shows an explicit "no key configured" prompt
instead of a canned answer.

| # | Brief category | Feature (where) | GenAI capability used | Why a non-AI system can't do this |
|---|---|---|---|---|
| 1 | **Navigation** | Assistant tab → "Navigation" quick chip (`tab-assistant`) | Open-ended reasoning over an unstructured user query (e.g. "I'm at Gate C with a stroller, how do I get to Section 114 and avoid stairs?") synthesized against structured venue POI data (`MOCK_POIS`) passed in the system prompt | A fixed if/else can only match pre-anticipated phrasings and can't compose a multi-step route that accounts for ad-hoc constraints (stroller, stairs, nearest elevator) it wasn't explicitly coded for |
| 2 | **Crowd management** | Ops tab → "Gate Congestion Forecaster" (`tab-ops`) | Narrative *prediction* generated by the LLM from a live-simulated occupancy time series across all gates (`MOCK_GATES`, drifted every few seconds) — the model reasons about trend + kickoff time-to-go and recommends a specific gate/window | A randomly-drifting progress bar only displays a number; it can't produce a comparative recommendation ("Gate D is trending down and will be clearer in ~15 min, route arriving fans there instead") — that requires reasoning across multiple time series at once |
| 3 | **Accessibility** | Assistant tab → "Accessibility" quick chip (`tab-assistant`) | Free-text description of an access need (mobility, sensory, hearing, etc.) synthesized against structured accessibility-feature data (`MOCK_ACCESSIBILITY`) into a personalized plan | Accessibility needs are highly individual combinations (e.g. "wheelchair + light sensitivity + traveling with a service animal"); a lookup table can only offer one generic page per need, not a composed plan for a combination it wasn't pre-scripted for |
| 4 | **Transportation** | Assistant tab → "Transport & Parking" quick chip (`tab-assistant`) | Synthesizes simulated transit/rideshare/parking feeds (`MOCK_TRANSIT`) with the user's stated origin, party size and timing into a specific recommendation | A static "transit pass included" bullet is content, not a decision; real synthesis is needed to trade off multiple simulated options against a user's specific situation |
| 5 | **Sustainability** | Sustainability tab → "Waste Sorting Identifier" (`tab-sustainability`) | Classifies an arbitrary, free-text item description into recycle/compost/landfill/hazardous **with a reasoned justification**, instead of matching a fixed keyword table | Real-world waste items are described in unlimited ways ("half-eaten nacho tray with plastic cheese cup") — a keyword table only covers items someone thought to hardcode; the LLM generalizes to unseen phrasing and explains *why* |
| 6 | **Multilingual assistance** | Header language selector, applied to every Assistant reply (`tab-assistant`) | The LLM is instructed (system prompt) to *generate* its answer natively in the selected language (including RTL Arabic), rather than looking up a pre-translated string | A phrasebook only covers pre-written strings; the assistant must generate a fresh answer in-language for whatever the fan actually asked, which no static translation table can do |
| 7 | **Operational intelligence** | Ops tab → "Generate Ops Brief" (`tab-ops`, staff-facing) | Synthesizes four simulated structured feeds at once — gate occupancy, incident log, weather, transit status — into a single narrative staff briefing with prioritized recommended actions | Combining four independent data sources into one coherent, prioritized narrative is exactly the "synthesizing multiple data sources into novel output" case a fixed template cannot do, since the relevant combination changes every time |
| 8 | **Real-time decision support** | Volunteer tab → "Incident Response Classifier" (`tab-volunteer`) | Takes a free-text incident description, classifies category/severity, **and** generates a specific next-action recommendation for that situation, replacing the old regex severity picker | A regex/keyword classifier can only bucket text it was written to expect and cannot produce a tailored, situational response recommendation for incidents it wasn't pre-coded to recognize |

### Labeling convention used throughout the UI
- **`SIMULATED DATA · REAL GENAI REASONING`** — the badge on Ops, Crowd, Sustainability and
  Assistant panels. The input feed (gate numbers, transit status, weather, POI list) is mock
  data because there is no live sensor/transit API in this prototype, but every response show is
  a real, uncached LLM completion over that data.
- **`FULLY SIMULATED`** — not currently used; nothing in this build fakes both the data and the
  reasoning. If a future feature can't get a real LLM call working, it must carry this label
  instead of being left unlabeled.

### Scope-creep review (Step 4)
- The old giant hero section, broadcast-style match ticker (fake live scores) and multi-team
  countdown were pure decoration with no tie to any of the 8 categories. The ticker has been
  **repurposed** into a live Ops ticker (gate/incident/weather status — ties to crowd management
  & operational intelligence) instead of removed outright, since a status ticker is still useful
  venue-staff UI. The countdown is kept only as a small header stat (kickoff context), not a
  hero centerpiece.
- Match schedule / host cities / ticket tiers are kept but demoted to a single **"Fan Info"**
  tab, explicitly labeled **"Static reference — no AI"**, so they are not miscounted as
  demonstrated GenAI capability.
